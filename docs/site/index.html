🧰 First Things First

The Go Tools Glossary

Before anything else, let's get comfortable with the commands and tools you'll see in this project. Every one of these is run from your terminal (the black-and-white text window). Think of this section as your cheat sheet — you can come back to it any time.

go build

Turns source code into a runnable program

What it does: Go source files end in .go, but your computer can't run them directly. Running go build reads your .go files, checks them against the Go language rules, and produces a single executable file (a "binary") — for example bin/queen on macOS/Linux or queen.exe on Windows.

Why we care: if something is syntactically wrong, Go will tell you before you try to run it. This catches typos, missing imports, and broken type usage in seconds.

The flag we use: -o bin/queen tells Go "put the resulting binary at bin/queen" instead of the default location. 

go build -o bin/queen ./pkg/queen

Read it like English: "Hey Go — turn everything under ./pkg/queen into an executable file called bin/queen."

go vet

A spell-checker for your code

What it does: Imagine writing an essay — a spell-checker catches typos your eye misses. go vet is the same idea for code. It scans your files looking for patterns that are technically valid Go, but are almost always bugs: shadow variables, unreachable code, suspicious printf format strings, and the like.

Important: go vet does not run your code. It only reads it. That means it's safe and very fast — a few hundred milliseconds even for big projects.

The ./... symbol you keep seeing: this is shorthand for "this folder, and every subfolder underneath it." So go vet ./... means "vet the entire project, all packages." 

go vet ./...

Read it like English: "Scan every Go file in this project for suspicious patterns."

go test ./...

Runs your automated tests

What it does: any file ending in _test.go contains automated tests. These tests are tiny programs that verify a piece of your code behaves correctly. go test runs all of them and prints a green PASS or red FAIL for each.

How to read a test file: 

// filename: parser_test.go
func TestParseValidInput(t *testing.T) {
    got := Parse("hello world")
    want := "Hello, World!"
    if got != want {
        t.Errorf("got %q, want %q", got, want)
    }
}

The test says: "Given the input hello world, the function Parse should return Hello, World!. If it doesn't, the test fails and tells us what went wrong."

Why it matters: with tests, you can change code confidently — if you break something, the test will tell you immediately instead of discovering it later in production.

go.mod & go.sum

Your project's "package list"

go.mod: the small file that describes your project's name, the Go version it needs, and the dependencies (third-party packages) it uses. Think of it like the ingredients list on a recipe card. 

// go.mod
module github.com/mcsr-ranked-bedrock
go 1.25.0
require modernc.org/sqlite v1.53.4

go.sum: an auto-generated file that contains cryptographic fingerprints for every dependency file. It's how Go proves the code it downloaded is exactly the code the project expects, and nothing was tampered with along the way. You almost never edit this by hand.

Analogy: if go.mod is the recipe card, go.sum is the tamper-evident seal on every ingredient.

typecheck

"Will this compile?" without producing a binary

What it means: Go is a statically-typed language. Every variable, function argument, and return value has a type (a string is text, an int is a whole number, a time.Time is a moment in time, etc.). The "typecheck" step is when Go verifies all your types line up — you passed a string where the function expected an int? Typecheck failed.

In our project: we use go build ./... as a typecheck in our Makefile. (When you only want typecheck without producing a real binary, you can use go build -o /dev/null ./... — but most people just run go build and ignore the produced binary if they don't need it.)

Why it matters: bugs that a "typecheck" catches would otherwise crash your program at runtime, often hours later, in production. Catching them up-front is enormously valuable.

SQLite (and "pure Go")

A tiny, file-based database

SQLite is a database that lives in one file on disk. There's no separate database server to install or run. That makes it perfect for self-contained apps like ours — the entire match history fits in a single file you can copy, e-mail, or back up trivially.

Traditional SQLite drivers in Go are "CGo" — they call into a C library. That means:

  • You need a C compiler installed to build the Go binary.
  • Cross-compiling for other operating systems is painful.

modernc.org/sqlite is what we use instead — it's a 100% Go translation of the SQLite C source. It's slower than the C version, but it removes the Go-to-C boundary entirely. For us, that's worth it: simple builds, no toolchain dependencies.

git & SSH push

Saving and sharing your work

git is a version control system — it tracks every change you make to your code, so you can undo mistakes, see history, and collaborate with others. A "repo" (short for repository) is a folder whose changes git is tracking.

The flow:

  1. git add -A — Stage every changed file to be saved.
  2. git commit -m "..." — Save a snapshot with a short message describing the change.
  3. git push — Upload that snapshot to a remote server (in our case, GitHub).

SSH is a way to authenticate to a remote server without typing a password — it uses a pair of cryptographic keys (a "private" key that stays on your computer and a "public" key that GitHub stores). When you run git push git@github.com:..., SSH proves it's really you.

HTTP / REST / JSON

How programs talk to each other

HTTP is the protocol your browser uses to fetch web pages. Our matchmaking service (the queen) listens for HTTP requests the same way a web server does — but instead of returning HTML for a browser, it returns JSON for another program (the worker) to read.

REST is a style of using HTTP where each URL represents a "resource":

  • POST /queue — "join the matchmaking queue"
  • GET /me — "tell me about myself"
  • GET /matches/42 — "give me match #42"

JSON (JavaScript Object Notation) is a text format for structured data. It looks like: 

{
  "xuid":  "alice-101",
  "elo":   1200,
  "match": { "id": 1, "seed": "..." }
}

Almost every computer-to-computer API in the world uses JSON. Easy for humans to read, easy for machines to parse.

🎯 What we're building

The Big Picture

Before we dive into files, let's talk about what this project actually does in plain English. Once you understand this, every file will make sense.

🏃 What's MCSR?

MCSR stands for Minecraft Speedrunning Community Ranked. Speedrunning is finishing a game as fast as possible. In Minecraft, that's usually beating the game (killing the Ender Dragon) while the game records exactly what happened and when.

"Ranked" means the speedrunning community wanted to compare players fairly — like a chess rating — so two players of similar skill are paired against each other.

🛏 What's "Bedrock"?

Minecraft comes in two main editions: Java Edition (the original, runs on PCs and Macs) and Bedrock Edition (cross-platform — phones, consoles, and Windows 10/11). They are different codebases with different mechanics, so they have separate ranked ladders.

This project — MCSR Ranked Bedrock Edition — is the ranked ladder for the Bedrock community. The Java equivalent already exists; this fills the gap.

🧩 What does this codebase do?

It is the central matchmaking service that powers that ranked ladder. When two players both want to play a ranked match, this service:

  1. Picks the matching pair based on rating.
  2. Picks a fair, surprise seed (the specific world layout they'll race in).
  3. Sends both players the match details so their games can launch automatically with identical worlds.
  4. Watches for the result (who won, who lost, was there a disconnect?) and updates each player's rating.

🪪 Phases so far

We are at phase 2 of 11. The foundation is built. What's left is mostly connecting it to real Minecraft server processes, real Xbox Live login, and shipping nice GUIs (Tauri launcher + Bedrock add-on) on top.

See the Phases section below for the full roadmap.

🏗 How the pieces talk

The Architecture

We have one central queen service running on a server, and a tiny worker helper running on each player's machine. They communicate over HTTP and a small set of message types.

Queen — central matchmaking, ratings, seeds, anti-cheat. One instance, public-internet facing.
Worker — small helper app on every player's machine. Connects to the queen, launches BDS locally, streams events back.
BDS — the actual Minecraft Bedrock server software. Spawns an identical world for both players using the seed the queen picked.

🧭 File-by-File

The Walkthrough

For each file, here's what it does, the key types and functions, and why it's structured the way it is. Read top-to-bottom for the deep dive, or jump in wherever you're curious.

cmd/seedgen/main.goTooling — build-time only

This isn't a server. It's a one-shot program you run once (or whenever the upstream seed list changes). It reads raw seed files from a build-time source bundle, runs them through pkg/shared/seeds for dedup and categorization, then writes both resources/seeds/categories/*.json and resources/seeds/manifest.json. The queen never runs this — it just consumes the outputs.

Key idea: the seed picking logic is the same code path used in production, so if the pipeline breaks, it breaks at build time (loud) instead of when matchmaking happens (silent disaster).

pkg/shared/seeds/*Shared library

The seed pipeline, isolated so it's easy to unit-test and reused by both the seed generator and the the queen at runtime.

  • manifest.go — turns the JSON manifest into typed Go structs (Manifest, Category, Seed) and exposes a Load(path) helper.
  • internal.go — the actual deduplication logic. Detects near-mirror seeds (they have the same first structure dig), buckets survivors by category (lone structure near 0,0 = its type), and Pick(category, rng) returns a random one.
  • manifest_test.go — 9 tests covering dedup correctness, category bucketing, deterministic picks, and error cases.
pkg/queen/main.goQueen entry point

When you run ./bin/queen, this is the file that runs first. It does five things, in order:

  1. Parse command-line flags (--addr, --db, --match-interval).
  2. Load config (overlaid on top of the env vars from internal/config).
  3. Set up the logger.
  4. Open the database and apply migrations.
  5. Start the HTTP server and the matchmaker tick. Wait here forever.

The boring-but-essential pattern: every service has a main.go that's just plumbing. The interesting logic lives in internal/.

pkg/queen/internal/config/config.goConfig loader

Reads all the configuration the queen needs. The pattern is "12-factor": config comes from environment variables, with command-line flags overriding them. So you can set MCSR_HTTP_ADDR=:9000 in your shell, or pass --addr=:9000 — whichever you prefer.

Loaded once at startup into a single Config struct and passed by pointer to everyone who needs it.

pkg/queen/internal/db/db.goDatabase bootstrap

The protocol for safely evolving a database over time is called migrations — small, numbered SQL scripts that are applied in order. Each one runs exactly once and is recorded in a tracking table so it's never re-run by accident.

db.go:

  1. Opens (or creates) the SQLite file at the configured path, or :memory: for tests.
  2. Runs a tiny embedded migration runner that reads every .sql in the migrations folder, in numeric order, and applies each one once.
  3. Returns a standard *sql.DB handle the rest of the program uses to execute queries.

Why INTEGER + unixepoch() for timestamps: we tried storing them as text first, but our pure-Go SQLite driver returns them in a quirky way that broke Go's automatic time scanning. Storing them as Unix seconds (a normal integer) and converting in our own helper was the simplest fix.

pkg/queen/internal/store/*Repository layer

A pattern: every SQL query the queen runs lives in this folder. HTTP handlers and the matchmaker never write SQL themselves — they call methods like store.Players().Upsert(...). That means:

  • If we ever want to swap SQLite for Postgres, only this folder changes.
  • If a query has a bug, you know exactly where to look.
  • It's easier to write unit tests against typed methods than against raw SQL strings.

scan_time.go is the unsung hero here. The pure-Go SQLite driver (modernc) has a quirk — INTEGER columns holding Unix timestamps come back sometimes as int64, sometimes as a digit string. This helper accepts both, so callers don't have to think about it.

pkg/queen/internal/matchmaker/*Matchmaking brain

The job: every so often, look at the queue, take the two people who've been waiting longest, give them a fair seed, and atomically create a matches row.

There's a background ticker (default: once per second) so idle entries eventually get paired, but also a manual Notify() hook the HTTP layer fires every time someone hits POST /queue. This way a player who joins doesn't have to wait up to a second for the next tick — pairing happens within milliseconds.

pkg/queen/internal/api/*HTTP handlers

Each file here is one or more HTTP endpoints. The routing table in router.go maps a URL pattern to a handler. Handlers:

  • Pull out headers (especially X-Player-XUID) via the auth middleware.
  • Sanity-check the request body.
  • Call into the store package to do the actual work.
  • Write a JSON response (or a useful HTTP error code if something went wrong).

Full endpoint list: POST /players, GET /players/{xuid}, POST /queue, DELETE /queue, DELETE /queue/{xuid}, GET /queue, GET /me, GET /me/match, GET /matches/{id}, GET /seeds/categories, GET /health.

pkg/worker/main.goPer-player helper (placeholder)

This is mostly a placeholder. In Phase 3 it will:

  1. Authenticate with the queen using the player's XUID.
  2. Poll the queen for assignments (or get them pushed).
  3. When matched, launch the Bedrock Dedicated Server locally with the queen's chosen seed.
  4. Watch that server process for events (player joins, world loaded, win, loss, disconnect, forfeit) and stream them back.
  5. Render a small overlay (Tauri, in Phase 10) so the player can see the match scoreboard and chat.
SQL migrationsDatabase schema

Four numbered scripts that build the schema step by step:

  • 0001_initial_schema.sql — creates a tiny schema_version table so the runner knows which migrations have already been applied.
  • 0002_players.sql — players table: xuid (Xbox ID, the primary key), gamertag, elo (default 1200), provisional (boolean, true until they've played enough matches to settle their true rating), and timestamp columns.
  • 0003_queue_entries.sql — queue table: one row per queued player, status of waiting or matched, plus the player ID and a timestamp.
  • 0004_matches.sql — matches table: one row per match created, with the seed, a status (pending → running → completed_* or forfeited), the two players, and timestamps.

The 11-Phase Roadmap

The full plan, distilled. Green chips are shipped; orange chips are pending; the small "💬" cards under Phase 9 are the chat-command sub-feature we added recently.

  1. ✓

    Phase 0 — Project Skeleton

    Set up the Go module, project layout, Makefile, and CI scaffolding.

  2. ✓

    Phase 1 — Seed Category Pipeline

    Wrangle the 88k+ FSG seeds into 6 categories with a typed manifest. That's the seedgen command.

  3. ✓

    Phase 2 — Accounts, Queue, Matchmaking

    Players, queue, pairings, ratings-ready. 10 endpoints. End-to-end test passed.

  4. 3

    Phase 3 — Local Worker Agent

    Real worker implementation that connects to the queen and runs BDS locally for each match.

  5. 4

    Phase 4 — Real BDS Startup

    Auto-detect or install BDS, generate correct server.properties, hand the chosen seed to BDS.

  6. 5

    Phase 5 — Result Events

    Winner / loser / cancel / forfeit handler with duplicate rejection and a robust state machine.

  7. 6

    Phase 6 — Ratings (Elo)

    Once a match ends, update both players' Elo. Promote past provisional after N matches. This is where /draw lands — both players' Elo moves toward each other.

  8. 7

    Phase 7 — Xbox Live Auth

    Replace the dev X-Player-XUID header with a real OAuth flow against Xbox Live.

  9. 8

    Phase 8 — Anti-Cheat

    Server-side timing analysis + cross-validation to catch obviously impossible runs.

  10. 9

    Phase 9 — MCBE Add-on

    A Bedrock add-on that wires in the chat commands and shows match UI in-game.

    💬 /forfeit

    Type it any time during a match. The match immediately ends, you lose, the other player wins. No confirmation needed.

    💬 /seedchange

    Requests a new, different seed. The other player sees the prompt and has 30 seconds to type /seedchange back to agree.
    If they agree, both BDS servers regenerate the world, the run timer resets, the timer of how many changes you've used ticks up.
    Maximum of 3 dual-agreed changes per match. Going over the limit simply ignores further requests.

    💬 /draw

    Same consent flow as /seedchange — the other player must agree within 30 seconds — but the result is the match ends with status completed_draw instead of a fresh seed. No Elo change.

    🗃️ Data model used by all three

    A new match_state_requests table tracks every pending proposal: id, match_id, request_type (seedchange | draw), requested_by, expires_at (now + 30s), status (pending | agreed | expired | rejected). The matches table gains a seed_change_count column to enforce the 3-per-match cap.

  11. 10

    Phase 10 — Tauri Overlay

    A small native window for the launcher that shows the chat commands UI, recent results, and live score.

  12. 11

    Phase 11 — Launch from Launcher

    Install the worker as a service, register a real Minecraft launch path so everything is one click.

📖 Speaking the language

General Glossary

Words you'll hear in this project that may be new. Each gets a plain-English definition and a short example.

📦 Module

Reusable bundle of code. In Go, a module is what your go.mod declares. Modules can be third-party ("dependencies") or your own ("internal packages").

🧩 Package

A single folder of .go files that share a namespace. Every file in that folder starts with package foo. Packages are the unit of reuse and visibility in Go.

🔧 Compile

To translate code from the language humans type (Go source) into the language computers execute (machine code). go build does this.

"Will it compile?" is a quick way to ask "is this code even valid Go?" — you don't actually need to run it to answer.

🚀 Binary

The file go build produces. A single, self-contained executable program. We put ours in bin/.

📝 Migration

A small SQL script that changes the structure (or sometimes contents) of a database. Each one is numbered so we always apply them in order, never twice. Migrations let you evolve a database safely over time instead of editing schema by hand.

🔒 Transaction

A group of database operations that either all succeed or none of them happen. Imagine moving money: you must subtract from one account AND add to another; if anything goes wrong in the middle, both should be undone. That's a transaction. We use them in CreateMatch so a match is never created without also flipping the queue rows.

🗄️ Repository pattern

A way of organizing code where all database access lives in a single dedicated layer (our pkg/queen/internal/store/). Higher layers (HTTP handlers, the matchmaker) call repository methods instead of writing SQL. Makes testing and swapping databases much easier.

🌐 HTTP / REST

The rules for how programs talk to each other over the network. "REST" is the style where URLs identify resources (/queue, /players/42) and HTTP verbs describe the action (POST, GET, DELETE).

🦎 SQLite

A database that lives in a single file on disk. No server to install. Perfect for self-contained apps. Our match history fits in one file you can copy or e-mail.

🧵 Goroutine

A "lightweight thread" in Go. Cheap to spawn — a typical Go program runs thousands at once without breaking a sweat. Our matchmaker runs one as a background ticker without blocking the HTTP server.

🏷️ Idempotent

A operation you can run any number of times and get the same result. Our JoinQueue is idempotent — joining twice doesn't put you in line twice; it just confirms you're already there.

🪙 Elo / Glicko

Rating systems. Elo is the classic chess rating: if you beat a stronger player, your rating jumps more than if you beat a weaker one. We'll use it (or its cousin Glicko-2) in Phase 6 to compute updated ratings after each match.

🛡️ Provisional rating

A starting rating used when we don't yet know a player's true skill. While "provisional", their results count more heavily so we find their real rating quickly.

🌱 Seed

A unique number that deterministically generates a Minecraft world. Same seed = same world. Two players in a ranked match are given the seed of a world we know is fair and interesting, so they race on identical terrain.

🧑‍💻 Worker

A small helper program that lives on the player's machine. It connects to the queen over the internet, launches BDS locally, and streams match events back.

🌍 BDS

Bedrock Dedicated Server — Mojang's official server software for Minecraft Bedrock Edition. The actual program that runs the multiplayer world.

🔀 TLS / HTTPS

Encrypted HTTP. Browsers show it as a padlock icon. We'll use it in production so XUIDs and match results can't be sniffed off the wire. In dev we run plain HTTP on localhost.

🪪 OAuth

The "log in with another service" pattern. We'll use Xbox Live OAuth in Phase 7 so players prove they really are who their gamertag says.

📡 Streaming

Sending data continuously rather than as a single big blob. Used for live match events so the queen sees updates within a second.
